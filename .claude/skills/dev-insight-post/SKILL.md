# 회고형 개발일지 포스트 작성

참고 원본: https://chictimin.github.io/blog/accidental-harness-engineering/

경험(삽질, 시행착오, 뒤늦은 깨달음)을 받아 1인칭 회고 글로 써서 `_posts/`에 저장한다. 핵심은 "완성된 결과물
자랑"이 아니라 "몰랐다가 알게 된 과정"이다.

이 블로그엔 두 결이 있다: 이모지 가득한 "Nani" 튜토리얼 시리즈(`author: Nani`, 초보자 How-to)와
`_posts/2026-07-10-welcome.md` 같은 담백한 회고. 이 스킬은 후자를 만든다 — "가이드/설명 써줘"라면 이
스킬이 아니다.

`examples/sample-post.md`는 이 스킬로 실제로 완성한 샘플이다. 구조·어조 참고용일 뿐, 그 안의 문장을
빈칸만 채워 재사용하지 않는다 — 매번 이번 소재의 실제 어휘로 새로 쓴다 (3단계 참고).

## Instructions

1. **소재 파악.** 이미 대화에 있으면 다시 묻지 않는다. 부족하면 짧게 확인한다: 무엇을 했나(기간·도구·행동) /
   처음엔 뭐라 생각했나 / 언제 다르게 보이기 시작했나(누가 이름 붙여줬나?) / 지금은 뭐라 부르고 왜 중요한가.

2. **구조 잡기.**
   1. 도입부 — 독자가 공감할 수 있도록 문제 제기 맥락을 제시한다(결과물로 보여줄 게 없다는 당혹감).
   2. 본문 — H2 헤딩으로 섹션을 구분하고, 깨달은 순서로 진행한다. 실제로 한 일들을 나열하다가, 이름·계기로
      그동안의 일이 하나로 묶이는 지점이 나오면 그게 전환점이자 제목의 근거가 된다. 필요하면 "한 일 ↔ 의미"
      2열 표를 더한다.
   3. 마치며 — 핵심 요약(담백하게 느낀 점 위주)과 다음 단계 제안으로 닫는다. 참고 링크가 있으면 마지막에
      덧붙인다.
   - 앱 자랑·기능 설명·튜토리얼로 새지 않는다. 소재가 짧으면 섹션을 줄여도 된다 — 개수를 채우는 게
     목적이 아니다.

3. **작성 스타일.**
   - 1인칭, 담백한 구어체로 쓴다.
   - 제목·소제목은 이번 소재의 실제 대사·장면으로 새로 짓는다 — 예시 문구를 빈칸만 채워 재사용하지
     않는다. 제목 후보를 2~3개 떠올려보고, 예시를 베낀 것 같은 후보는 버린다.
   - 내적 독백은 그 순간 실제 떠오른 말을 따옴표로 인용한다.
   - 자조적 유머와 솔직한 고백은 아끼되 과한 자기비하는 넣지 않는다.
   - 문단은 3~4문장을 넘으면 쪼갠다.
   - 도구·모델·명령어는 실명 그대로 쓴다.
   - 각주는 흐름을 안 끊는 부가 설명에만(`[^1]` ... `[^1]: 설명`).
   - 번역투·과장된 수식어·잦은 볼드체는 피한다.

4. **코드 블록.** 항상 언어를 지정한다(`js`, `py`, `rust`, `css`, `html`, `bash` 등). 타이틀이 필요하면
   `언어 title="제목"` 형식을, 문맥상 명확하면 생략한다.

5. **이미지 자리 표시 + 추천.** 이미지를 직접 만들거나 다운로드하는 기능은 없다 — 위치와 "무엇을 넣을지"를
   구체적으로 남긴다.
   ```markdown
   [이미지 자리 — 스크린샷 추천: OO 명령 실행 직후 화면. alt: "..."]
   [이미지 자리 — 밈 추천: "..." 상황에 어울리는 밈. 검색 키워드: "..."]
   ```
   - **스크린샷**: 사용자가 실제로 그 작업을 하며 봤을 화면을 구체적으로 짚는다.
   - **밈**: 담아야 할 감정/상황을 먼저 설명하고, 검색 키워드를 1~2개 예시로 제안한다(실제 이미지는
     만들지 않는다).
   - 보통 들어가며 끝(밈 1개), 실제 화면이 있는 항목(스크린샷 1~2개), 마치며 직전(선택, 1개) 순으로
     배치한다. 다 쓴 뒤 이미지 자리를 목록으로 정리해 사용자에게 보여준다.

6. **분량·파일·front matter.** front matter는 화면에 그대로 보이는 게 아니라, 테마가 읽어서 제목·날짜·
   태그 UI로 렌더링하는 메타데이터다 — 그래서 반드시 front matter로 적는다.
   - 목표 분량 3,000~4,000자. 소재가 얕으면 짧아도 된다 — 억지로 늘리지 않는다.
   - 파일명: `_posts/YYYY-MM-DD-NN-슬러그.md` (오늘 날짜 + 같은 날 다음 번호). 저장 전 번호 중복을
     확인한다.
   - front matter:
     ```yaml
     ---
     title: "글 제목"
     tags:
       - (7단계에서 고른 키워드 3~5개)
     date: YYYY-MM-DD
     description: "200자 이내 SEO 설명"
     ---
     ```
   - `description`은 핵심 깨달음/질문을 한 문장으로 압축하고 키워드를 한두 개 자연스럽게 포함한다 —
     Nani 시리즈처럼 내용을 미리 요약하지 않고, 궁금증을 유발하는 문장으로 쓴다.
   - `author` 필드는 Nani 시리즈 전용이므로 넣지 않는다.

7. **SEO 태그 추천.** 검색될 법한 구체적 키워드와 상위 개념 키워드를 섞어 3~5개로 구성한다(예:
   "하네스엔지니어링"(핵심) + "AI에이전트"(상위) + "회고"(형식)). `_posts/`의 기존 표기 관례를 따른다 —
   영문 기술명은 붙여쓰기(`GitHubActions`, `SQLite`), 한국어 개념어는 그대로(`회고`, `세션`). 3단계에서
   정한 제목에 핵심 개념이 이미 들어 있는지만 확인하고, SEO 때문에 제목을 다시 정형화하지 않는다.
   - front matter의 `tags`에 넣는 것과 똑같은 태그를, 본문 맨 끝(참고 자료 다음)에 눈에 보이는 배지로도
     한 번 더 적는다(예: `` `태그1` `태그2` `태그3` ``). front matter 태그는 사이트의 태그 아카이브
     기능을 위한 것이고, 본문 하단 배지는 글을 읽는 사람이 바로 볼 수 있게 하는 것이라 둘 다 필요하다.

8. **마무리 전 점검.**
   - [ ] 3단 구조가 뚜렷한가
     - 도입부엔 당혹감, 마무리엔 실용적 통찰 + 메타 인식
   - [ ] 문체가 맞는가
     - 제목·소제목을 이번 소재로 새로 썼고(예시 베끼지 않음)
     - 내적 독백이 인용돼 있고
     - Nani 톤이 안 섞였는가
   - [ ] 형식이 맞는가
     - 문단이 짧고, 코드 블록마다 언어가 지정됨
     - 파일명·front matter(tags 3~5개·description 200자 이내)·본문 하단 태그 배지가 규칙대로임
     - 이미지 자리별 구체적 추천이 갖춰짐
   - 완료 후 파일 경로를 알려주고, 고칠 부분이 있는지 사용자에게 확인한다.
