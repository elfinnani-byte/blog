# 회고형 개발일지 포스트 작성

참고 원본: https://chictimin.github.io/blog/accidental-harness-engineering/

경험(삽질, 시행착오, 뒤늦은 깨달음)을 받아 1인칭 회고 글로 써서 `_posts/`에 저장한다. 핵심은 "완성된 결과물
자랑"이 아니라 "몰랐다가 알게 된 과정"이다.

이 블로그엔 두 결이 있다: 이모지 가득한 "Nani" 튜토리얼 시리즈(`author: Nani`, 초보자 How-to)와
`_posts/2026-07-10-welcome.md` 같은 담백한 회고. 이 스킬은 후자를 만든다 — "가이드/설명 써줘"라면 이
스킬이 아니다.

`examples/sample-post.md`는 이 스킬로 실제로 완성한 샘플이다. 구조·어조 참고용일 뿐, 그 안의 문장을
빈칸만 채워 재사용하지 않는다 — 매번 이번 소재의 실제 어휘로 새로 쓴다 (3단계 참고).

## Instructions

1. **소재 파악.** 이미 대화에 있으면 다시 묻지 않는다. 부족하면 짧게 확인한다: 무엇을 했나(기간·도구·행동) /
   처음엔 뭐라 생각했나 / 언제 다르게 보이기 시작했나(누가 이름 붙여줬나?) / 지금은 뭐라 부르고 왜 중요한가.

2. **3단 구조로 뼈대 잡기.** 문제 제기(결과물로 보여줄 게 없다는 당혹감) → 깨달음(이름·계기로 그동안 한
   일이 하나로 묶이는 전환점, 제목의 근거) → 의미 확장(왜 중요한지 담백하게). 앱 자랑·기능 설명·튜토리얼로
   새지 않는다.

3. **제목·소제목을 이번 소재로 새로 쓰기.** 예시는 "이 정서를 이렇게도 쓸 수 있다"는 하나의 사례이지,
   빈칸 채우는 서식이 아니다. 실제 있었던 대사·장면을 살리면 자연히 예시와 다른 문장이 나온다. 제목
   후보를 2~3개 떠올려보고, 예시를 베낀 것 같은 후보는 버린다.
   - 기본 섹션 순서: (H1 제목) → 들어가며 → 깨달음의 계기 → 한 일들(번호 매긴 소제목) →
     이게 왜 OOO인가(선택: "한 일 ↔ 의미" 2열 표) → 왜 지금 주목할까(선택) → 마치며 → 참고 자료(선택).
   - 소재가 짧으면 3~4개 섹션으로 줄여도 된다 — 개수를 채우는 게 목적이 아니다.

4. **문체.** 1인칭, 담백한 구어체. 내적 독백은 그 순간 실제 떠오른 말을 따옴표로 인용한다. 자조적 유머와
   솔직한 고백은 아끼되 과한 자기비하는 넣지 않는다. 문단은 3~4문장을 넘으면 쪼갠다. 도구·모델·명령어는
   실명 그대로 쓴다. 각주는 흐름을 안 끊는 부가 설명에만(`[^1]` ... `[^1]: 설명`). 번역투·과장된
   수식어·잦은 볼드체는 피한다.

5. **코드 블록.** 항상 언어를 지정한다(`js`, `py`, `rust`, `css`, `html`, `bash` 등). 타이틀이 필요하면
   `언어 title="제목"` 형식을, 문맥상 명확하면 생략한다.

6. **이미지 자리 표시 + 추천.** 이미지를 직접 만들거나 다운로드하는 기능은 없다 — 위치와 "무엇을 넣을지"를
   구체적으로 남긴다.
   ```markdown
   [이미지 자리 — 스크린샷 추천: OO 명령 실행 직후 화면. alt: "..."]
   [이미지 자리 — 밈 추천: "..." 상황에 어울리는 밈. 검색 키워드: "..."]
   ```
   - **스크린샷**: 사용자가 실제로 그 작업을 하며 봤을 화면을 구체적으로 짚는다.
   - **밈**: 담아야 할 감정/상황을 먼저 설명하고, 검색 키워드를 1~2개 예시로 제안한다(실제 이미지는
     만들지 않는다).
   - 보통 들어가며 끝(밈 1개), 실제 화면이 있는 항목(스크린샷 1~2개), 마치며 직전(선택, 1개) 순으로
     배치한다. 다 쓴 뒤 이미지 자리를 목록으로 정리해 사용자에게 보여준다.

7. **분량·파일·front matter.**
   - 목표 분량 3,000~4,000자. 소재가 얕으면 짧아도 된다 — 억지로 늘리지 않는다.
   - 파일명: `_posts/YYYY-MM-DD-NN-슬러그.md` (오늘 날짜 + 같은 날 다음 번호). 저장 전 번호 중복을
     확인한다.
   - front matter:
     ```yaml
     ---
     title: "글 제목"
     tags:
       - (8단계에서 고른 키워드 3~5개)
     date: YYYY-MM-DD
     description: "100~120자 SEO 설명"
     ---
     ```
   - `description`은 핵심 깨달음/질문을 한 문장으로 압축하고 키워드를 한두 개 자연스럽게 포함한다 —
     Nani 시리즈처럼 내용을 미리 요약하지 않고, 궁금증을 유발하는 문장으로 쓴다.
   - `author` 필드는 Nani 시리즈 전용이므로 넣지 않는다.

8. **SEO 태그 추천.** 검색될 법한 구체적 키워드와 상위 개념 키워드를 섞어 3~5개로 구성한다(예:
   "하네스엔지니어링"(핵심) + "AI에이전트"(상위) + "회고"(형식)). `_posts/`의 기존 표기 관례를 따른다 —
   영문 기술명은 붙여쓰기(`GitHubActions`, `SQLite`), 한국어 개념어는 그대로(`회고`, `세션`). 3단계에서
   정한 제목에 핵심 개념이 이미 들어 있는지만 확인하고, SEO 때문에 제목을 다시 정형화하지 않는다. 다 쓴
   뒤 추천 태그와 이유를 한 줄씩 보여준다.

9. **마무리 전 점검.**
   - [ ] 3단 구조가 뚜렷한가
     - 도입부엔 당혹감, 마무리엔 실용적 통찰 + 메타 인식
   - [ ] 문체가 맞는가
     - 제목·소제목을 이번 소재로 새로 썼고(예시 베끼지 않음)
     - 내적 독백이 인용돼 있고
     - Nani 톤이 안 섞였는가
   - [ ] 형식이 맞는가
     - 문단이 짧고, 코드 블록마다 언어가 지정됨
     - front matter(파일명·tags 3~5개·description 100~120자)가 규칙대로임
     - 이미지 자리별 구체적 추천이 갖춰짐
   - 완료 후 파일 경로를 알려주고, 고칠 부분이 있는지 사용자에게 확인한다.
