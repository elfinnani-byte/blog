# [바이브 코딩 3편] 가짜 데이터로 화면을 먼저 그린다? 비개발자도 쉽게 이해하는 API와 목업 데이터의 비밀
*(부제: 식당 비유로 끝내는 프론트엔드-백엔드 대화법)*

"화면(프론트엔드)은 다 그렸는데, 여기에 들어갈 진짜 데이터는 어디서 오는 걸까?"

지난 1, 2편을 통해 우리는 프레임워크의 개념을 잡고, AI와 소통하며 든든한 설계도인 PRD(제품 요구사항 정의서)까지 완성했습니다. 

이제 드디어 완성된 화면에 진짜 정보를 채워 넣고, 이 정보가 프론트엔드와 백엔드 사이를 어떻게 오가는지 배울 차례입니다. 어렵게만 느껴졌던 데이터 통신의 핵심 원리들을 맛있는 식당 비유와 함께 한 번에 정리해 드릴게요!

## 1. 프론트엔드와 백엔드는 어떻게 대화할까? — 식당 비유의 완성판

우리가 웹사이트에서 버튼을 누르면 눈에 보이지 않는 뒷단(백엔드)에서 무수히 많은 데이터가 움직입니다. 이 복잡한 과정을 '식당'으로 비유하면 아주 명확해집니다.



* **홀 (프론트엔드):** 손님이 앉아서 예쁜 인테리어를 구경하고 메뉴판을 보는 공간입니다. 
* **주방 (백엔드):** 손님 눈에는 보이지 않지만, 실제로 요리(데이터 처리)가 일어나는 곳입니다. 이 주방 안에는 두 주인공이 있습니다.
  * **요리사 (서버 프로그램):** 손님의 주문을 받아 직접 요리를 하고 머리를 쓰는 브레인입니다.
  * **식료품 창고 (데이터베이스, DB):** 온갖 재료(회원 정보, 상품 목록)가 신선하고 체계적으로 보관된 저장소입니다.
* **메뉴판과 점원 (API):** 손님이 주방에 함부로 들어가지 못하게 통제하면서, **"정해진 메뉴(규칙)대로만 주문해 주세요"**라고 약속하는 소통 창구입니다.
* **주문서 양식 (JSON):** 점원이 주방에 주문을 전달할 때 쓰는 **공통 언어**입니다. 누구나 알아보기 쉽게 정돈된 텍스트 형식이죠.
* **배달 수단 (HTTP):** 주문서를 들고 홀과 주방을 바쁘게 오가는 **점원의 발걸음**입니다. 정보를 가져올 때는 `GET`, 정보를 저장할 때는 `POST`라는 발걸음 방식을 씁니다.

> **💡 한 줄 요약:**
> "손님(프론트엔드)이 **API(메뉴판)**를 보고 주문서를 작성하면, 점원이 이 내용을 **JSON(주문서)** 형식으로 이쁘게 적어 **HTTP(발걸음)**라는 이동 수단을 타고 주방(백엔드 서버)으로 배달을 갑니다."

## 2. 목업 데이터 vs 더미 데이터 — 가짜 데이터의 미세한 차이

아직 주방(백엔드)이 공사 중이라 완성되지 않았는데, 홀(프론트엔드) 인테리어가 먼저 끝났다면 어떻게 해야 할까요? 음식이 나올 때까지 손님에게 빈 접시만 보여줄 수는 없겠죠. 

이때 사용하는 것이 바로 **'가짜 데이터'**입니다. 실무에서는 이 가짜 데이터를 두 가지 용어로 엄격하게 구분해서 사용합니다.



### 🎨 목업 데이터 (Mock Data) = "홀(화면) 전시용 모형 음식"
* **역할:** 백엔드가 아직 없어서 진짜 데이터를 불러올 수 없을 때, **"진짜 데이터가 오면 이런 느낌으로 화면에 그려질 거야"**를 보여주기 위해 프론트엔드 개발자가 가짜로 꾸며둔 정교한 데이터입니다.
* **특징:** 양은 적지만 실제 서비스와 똑같이 정교한 구조(상품명, 진짜 같은 이미지 주소, 정확한 가격 등)를 가집니다.
* **예시 (JSON 형태):**

    {
      "id": 1,
      "name": "딸기 생크림 케이크",
      "price": 32000,
      "category": "케이크",
      "image": "[https://example.com/strawberry_cake.jpg](https://example.com/strawberry_cake.jpg)",
      "soldOut": false
    }
  ]
