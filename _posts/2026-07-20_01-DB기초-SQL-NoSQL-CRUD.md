## 1. 파일 저장의 한계와 데이터베이스(DB)의 필요성

처음 프로젝트를 시작할 때 '그냥 텍스트 파일이나 엑셀에 데이터를 저장하면 안 되나?'라는 생각이 들 수 있습니다. 하지만 사용자와 데이터가 늘어나면 파일 방식은 금방 한계에 부딪힙니다.

* **동시 접속 문제**: 여러 사용자가 동시에 글을 쓰거나 수정하면 파일이 깨질 수 있습니다.
* **검색 및 성능 한계**: 데이터가 수십만 건이 넘어가면 원하는 정보를 찾는 데 엄청난 시간이 걸립니다.
* **보안 및 정교한 관리**: 데이터의 접근 권한을 세밀하게 나누기 어렵습니다.

이를 해결하기 위해 등장한 것이 바로 **데이터베이스(DB)**입니다. DB는 대량의 데이터를 안전하고 빠르고 체계적으로 관리해 주는 전문 데이터 창고입니다.

---

## 2. 관계형 DB(SQL) vs 비관계형 DB(NoSQL) 한눈에 비교하기

데이터베이스는 크게 **SQL**과 **NoSQL** 두 가지 스타일로 나뉩니다.

| 구분 | SQL (관계형 DB) | NoSQL (비관계형 DB) |
| :--- | :--- | :--- |
| **데이터 구조** | 엑셀처럼 정해진 **표(테이블)** 형태 | JSON 문서, Key-Value 등 **유연한 형태** |
| **규칙 (스키마)** | **엄격함** (사전에 컬럼/타입 정의 필요) | **자유로움** (데이터 구조를 언제든 변경 가능) |
| **강점** | 데이터의 **정확성 및 일관성** (금융, 주문 등) | **대용량 처리** 및 빠른 확장성 (SNS, 로그 등) |
| **대표 종류** | MySQL, PostgreSQL, Oracle, SQLite | MongoDB, Redis, Cassandra, DynamoDB |

> 💡 **한 줄 요약**  
> * **SQL**: 데이터의 정확성과 규칙이 중요한 서비스(회원/결제 등)에 적합  
> * **NoSQL**: 데이터 구조가 자주 바뀌거나 엄청난 양의 실시간 데이터(SNS 피드 등)에 적합

---

## 3. 표에게 말을 거는 언어, SQL과 CRUD 개념

SQL(Structured Query Language)은 쉽게 말해 **"표(테이블) 형태의 DB와 대화하는 언어"**입니다.

우리가 서비스를 이용할 때 일어나는 거의 모든 데이터 작업은 **CRUD**라는 4가지 기본 기능으로 이루어집니다.

* **C (Create / 생성)**: 새로운 데이터를 저장  
  * *"표야, 신규 회원 정보 한 줄 추가해 줘!"* (`INSERT`)
* **R (Read / 조회)**: 저장된 데이터를 검색  
  * *"표야, 오늘 가입한 회원 목록 보여줘!"* (`SELECT`)
* **U (Update / 수정)**: 기존 정보를 변경  
  * *"표야, 이 회원의 프로필 사진을 바꿔줘!"* (`UPDATE`)
* **D (Delete / 삭제)**: 데이터를 삭제  
  * *"표야, 탈퇴한 회원 정보를 삭제해 줘!"* (`DELETE`)
