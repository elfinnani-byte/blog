## 1. 로그인 인증 방식: 세션(Session) vs 토큰(JWT)

로그인 후 "내가 로그인되어 있다"는 상태를 유지하는 대표적인 두 가지 방법이 있습니다. 바로 **서버가 기억하는 세션**과 **사용자가 들고 다니는 토큰**입니다.

| 구분 | 세션 (Session) | 토큰 (Token, JWT) |
| :--- | :--- | :--- |
| **상태 저장 위치** | **서버** 메모리 / DB | **클라이언트** (브라우저, 앱) |
| **원리** | 서버가 세션 ID(번호표)만 브라우저 쿠키로 내줌 | 서버 도장이 찍힌 위조 방지 증명서를 발급 |
| **서버 부담** | 사용자가 늘수록 서버 메모리 부담 증가 | 서버가 기록을 두지 않아 과부하가 적음 |
| **확장성** | 서버를 여러 대 늘릴 때 세션 공유 설정 필요 | 서버 확장(Scale-out)이나 다중 서비스 연동에 유리 |
| **보안 제어** | 서버에서 세션을 삭제해 **즉시 강제 로그아웃** 가능 | 한 번 내준 토큰은 만료 전까지 **즉시 무효화 어려움** |

> 💡 **한 줄 요약**  
> * **단순 웹 & 강력한 통제 필요** ➡️ **세션** 방식  
> * **모바일 앱, API 서버, 대규모 확장 필요** ➡️ **토큰(JWT)** 방식  
> *(직접 구현하기보다 Supabase Auth나 Auth.js 같은 검증된 라이브러리를 쓰는 것이 안전합니다!)*

## 2. 웹 데이터와 DB를 지키는 보안 방어법

### ① 프론트엔드 XSS(Cross-Site Scripting) 방지
사용자 입력창에 악성 스크립트(`<script>...`)를 넣어 실행시키는 공격을 막으려면, 입력을 화면에 뿌릴 때 반드시 **소독(Sanitizing) 및 이스케이프** 처리를 해야 합니다.
* **HttpOnly Cookie**: 보안 토큰은 자바스크립트로 접근할 수 없는 `HttpOnly` 속성의 쿠키로 보관합니다.

### ② "구글 드라이브 시트를 DB로 쓰면 안 될까?"
구글 시트도 DB처럼 활용할 수 있지만, 실제 서비스 운영 시 분명한 **보안 한계**가 존재합니다.
* **API 키 노출 위험**: 브라우저에서 직접 시트에 접근하면 API 키가 노출됩니다.
* **행 단위 보안(Row-Level Security) 불가능**: 시트 전체 권한만 줄 수 있어 다른 사람의 개인정보까지 노출될 수 있습니다.
* **동시 쓰기 제한**: 접속자가 몰리면 데이터 충돌 및 응답 지연이 발생합니다.

## 3. 상황별 무료 DB 고르기 (SQLite vs Supabase)

웹 개발이나 바이브 코딩 시 상황에 맞춰 DB를 선택하는 것이 좋습니다.

* **SQLite (로컬 전용 DB)**:
  * 내 컴퓨터에 `.db` 파일 하나로 저장되는 아주 가벼운 DB입니다.
  * 인터넷 연결 없이 오프라인에서 빠르게 개발 및 테스트할 때 적합합니다.
* **Supabase (클라우드 Postgres DB)**:
  * 무료로 클라우드 DB를 제공하며, 회원가입/로그인 인증 기능 및 **RLS(행 수준 보안)** 기능이 내장되어 있습니다.
  * 남들에게 내 웹 앱을 링크로 보여주거나 실제 서비스로 배포할 때 최고의 선택입니다.
