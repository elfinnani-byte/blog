오늘 하려던 일은 단순했다. chictimin님의 블로그 글을 보여주면서, "이런 식으로 글을 쓰는 스킬 하나 만들고 싶다"고 부탁했다. SKILL.md 파일 하나에 문체 가이드를 정리해서 넣으면 끝나는 일이라고 생각했다.

그런데 description 한 줄을 몇 번이나 다시 썼다. 이 스킬이 정확히 언제 작동해야 하는지, 이 블로그의 다른 튜토리얼 톤과는 어떻게 구분되는지 적다 보니 문장이 계속 길어졌다. 파일도 하나가 아니라 두 개가 됐고, 이미지 자리를 어떻게 표시할지, 태그는 어떤 기준으로 고를지까지 정하고 있었다. 근데 그게 뭐 하는 스킬인데, 라고 누가 물으면 "블로그 글 쓰는 스킬"이라는 대답으로는 뭔가 부족했다.

[이미지 자리 — 밈 추천: "한 줄만 고치면 될 줄 알았는데 반나절이 지난" 상황에 어울리는 밈. 검색 키워드 예시: "시간 순삭 밈", "이걸 왜 이렇게 오래 하고 있지 밈"]

## 가이드로 점검해달라고 했던 그 순간

만든 파일이 제대로 됐는지 daleseo.com의 SKILL.md 작성 가이드에 맞춰 점검해달라고 부탁했다. 그 가이드에 정리된 항목들 — description 작성 규칙, 파일 길이 제한, 참조 파일 구조, 자유도(degrees of freedom) 조절[^1] — 을 하나씩 짚어보다가 알았다. 내가 반나절 동안 헤맨 일들이 전부, 이미 원칙까지 정리되어 있는 하나의 작업 분야였다는 것을.

## 그 사이에 실제로 손댄 것들

### 1. description 문구를 계속 다듬기

처음 쓴 description은 "회고형 블로그 글을 써준다" 정도였다. 트리거 조건과 다른 톤과의 경계까지 명시하려니 문장이 계속 늘어났다.

### 2. 저작권 선 안에서 예시 만들기

원문 문단을 그대로 옮겨달라는 요청을 받았을 때는, 15단어 이내의 짧은 인용과 구조 설명으로 대체했다. 참고 글을 통째로 복제하는 건 스킬 밖의 일이었다.

### 3. 이미지 자리를 표시하고 추천하는 로직 만들기

스크린샷 자리와 밈 자리를 구분하고, 실제로 이미지를 만들 수는 없으니 "무엇을 캡처하면 좋을지"까지 구체적으로 짚어주는 방식으로 설계했다.

[이미지 자리 — 스크린샷 추천: `.claude/skills/dev-insight-post/` 폴더가 `SKILL.md`와 `examples/`로 나뉜 모습을 보여주는 파일 탐색기 또는 `ls` 결과 캡처. alt 텍스트 예시: "SKILL.md와 examples 폴더로 나뉜 스킬 디렉토리 구조"]

### 4. 가이드 기준으로 점검하고 파일 쪼개기

daleseo.com 가이드에 맞춰 이름을 짧게 유지하고, 긴 설명 부분을 `examples/` 폴더로 분리했다.

### 5. SEO 태그 추천 로직 추가하기

태그도 아무거나 나열하는 게 아니라, 구체적 키워드와 상위 개념 키워드를 섞고 기존 포스트의 표기 관례를 따르게 했다.

## 손댄 일들을 다시 보면

| 한 일 | 의미 |
|---|---|
| description 반복해서 다듬기 | 트리거 정확도 설계 |
| 저작권 고려한 예시 작성 | 콘텐츠 정책 준수 |
| 이미지 자리 + 추천 로직 | 자유도 조절 |
| 파일 분리 | 컨텍스트·토큰 예산 관리 |
| SEO 태그 추천 로직 | 실사용 시나리오 확장 |

## 마치며

모델에게 정확히 뭘 시킬지 정하는 일, 지침의 자유도를 조절하는 일, 콘텐츠 정책 안에서 예시를 만드는 일, 컨텍스트 예산을 관리하는 일. 반나절 동안 한 건 그냥 "블로그 글 쓰는 법을 알려주는 것"이 아니라, 도구 자체를 설계하는 별개의 작업이었다.

재밌는 건, 이 글도 그렇게 만든 스킬의 첫 산출물이라는 점이다. description 한 줄 고치는 데 반나절이 걸린 이유를, 반나절짜리 결과물로 설명하고 있는 셈이다.
